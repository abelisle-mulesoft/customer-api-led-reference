## Overview

The API Security Traits Library is a reusable RAML 1.0 library a reusable RAML 1.0 library that defines security traits for use across APIs within the Customer API-led reference architecture. It provides standardized traits for enforcing authentication and authorization requirements, such as Client ID Enforcement, ensuring consistent security implementation across APIs.

By centralizing security traits in a shared library, API designers can apply consistent security controls across multiple APIs while avoiding duplication and divergence. This approach promotes uniform enforcement of access policies and simplifies API governance.

## Mental Model

Think of these libraries as defining: who can call the API, what happens, and how responses are structured.

- **Who can call the API** → defined by the API Security Traits Library  
- **What happens when a request is processed** → defined by the API Response Traits Library  
- **How responses are structured** → defined by the Common Response Types Library  

Together, they ensure consistent access control, behavior, and response structure across all APIs.

## Design Principles

**1. Centralized security enforcement:** Security requirements are defined once and reused across APIs to ensure consistency.

**2. API gateway alignment:** Traits are designed to align with policies enforced through Anypoint API Manager and the MuleSoft API gateway.

**3. Reuse through libraries:** Security traits are packaged as a RAML 1.0 library to enable consistent reuse across APIs.

**4. Separation of concerns:** Security enforcement is defined declaratively in API specifications while runtime enforcement is handled by the API gateway.

**5. Consistency across APIs:** Using shared traits ensures that all APIs implement authentication and authorization in a consistent manner.

## Library Contents

The RAML library defines reusable traits for applying security requirements to API methods.

Common traits include:

- `client-id-required` -- Enforces Client ID and Client Secret headers for API access.
- Additional traits may be added over time to support evolving security requirements.

## Typical Usage

This library is designed for use across all API layers.

- System APIs use these traits to enforce access to underlying systems.
- Process APIs use these traits to secure orchestration logic.
- Experience APIs use these traits to protect consumer-facing endpoints.

## Relationship to Other Assets

This library defines standardized access control requirements for APIs.

It works in conjunction with:

- [API Response Traits Library](https://anypoint.mulesoft.com/exchange/16339d91-3161-43b4-bef8-7e3fbf0af852/lib-response-traits): defines standard HTTP response behavior for both successful and error scenarios.
- [Common Response Types Library](https://anypoint.mulesoft.com/exchange/16339d91-3161-43b4-bef8-7e3fbf0af852/lib-common-responses): provides the standard response envelope used for error responses generated by security enforcement.

Together, these libraries establish a consistent model where:

- Access control is defined in this library
- Response behavior is defined in response traits
- Response structure is defined in the common response model
