## Overview

The API Security Schemes Library is a reusable RAML 1.0 library that defines security schemes for use across APIs within the Customer API-led reference architecture. It provides reusable security schemes for declaring authentication and authorization requirements, promoting a consistent security model across APIs.

By centralizing security schemes in a shared library, API designers can apply consistent security controls across multiple APIs while avoiding duplication and divergence. This approach promotes uniform enforcement of access policies and simplifies API governance.

## Mental Model

Think of these libraries as defining: who can call the API, what happens, and how responses are structured.

- **Who can call the API** → defined by the API Security Schemes Library
- **What happens when a request is processed** → defined by the API Response Traits Library
- **How responses are structured** → defined by the Common Response Types Library

Together, they promote consistent access control, behavior, and response structure across APIs that adopt the shared model.

## Design Principles

**1. Centralized security enforcement:** Security requirements are defined once and reused across APIs to ensure consistency.

**2. API gateway alignment:** Security schemes are designed to align with policies enforced through Anypoint API Manager and the MuleSoft API gateway.

**3. Reuse through libraries:** Security schemes are packaged as a RAML 1.0 library to enable consistent reuse across APIs.

**4. Separation of concerns:** Security enforcement is defined declaratively in API specifications while runtime enforcement is handled by the API gateway.

**5. Consistency across APIs:** Using shared security schemes promotes a consistent way to declare authentication and authorization requirements across APIs.

## Library Contents

The RAML library defines reusable security schemes for applying security requirements to API methods. Common security schemes include:

- `client-id-enforcement` -- Enforces Client ID and Client Secret headers for API access.
- Additional security schemes may be added over time to support evolving security requirements.

## Typical Usage

This library can be used across API layers.

- System APIs use these security schemes to enforce access to underlying systems.
- Process APIs use these security schemes to secure orchestration logic.
- Experience APIs use these security schemes to protect consumer-facing endpoints.

## Relationship to Other Assets

This library defines standardized access control requirements for APIs. It works in conjunction with:

- [API Response Traits Library](https://anypoint.mulesoft.com/exchange/16339d91-3161-43b4-bef8-7e3fbf0af852/lib-response-traits): defines reusable access control requirements for APIs.
- [Common Response Types Library](https://anypoint.mulesoft.com/exchange/16339d91-3161-43b4-bef8-7e3fbf0af852/lib-common-responses): provides the shared response model commonly used for error responses generated by security enforcement.

Together, these libraries establish a consistent model where:

- Access control is defined in this library
- Response behavior is defined in response security schemes
- Response structure is defined in the common response model
